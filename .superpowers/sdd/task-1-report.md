## Deviations from the literal plan text (and why)

1. **Removed `import './index.css'` from `src/main.tsx`.** The scaffold's default
   `src/index.css` is Vite/React demo boilerplate (sets its own `--bg`, `--text`, `--accent`
   CSS variables, `#root` width/flex/text-align rules, etc.) that directly collides with
   and would visually fight the project's own `theme.css` tokens. Since Task 1's file list
   doesn't mention `src/index.css` or `src/App.css` at all, and the target `App.tsx` only
   imports `./styles/theme.css`, I did not copy `src/index.css`/`src/App.css`/the demo
   assets (`react.svg`, `vite.svg`, `hero.png`) into the project at all, and dropped the
   now-dangling import from `main.tsx`. Nothing in the codebase references them.

2. **Added `"strict": true` to `tsconfig.app.json` and `tsconfig.node.json`.** The current
   (2026) `create-vite@latest react-ts` template ships *without* `strict: true` in its
   generated tsconfig — it relies on a long list of individual flags
   (`noUnusedLocals`, `verbatimModuleSyntax`, etc.) but not full strict mode. Global
   Constraints explicitly require "TypeScript strict mode on", so I added it to both
   the app and node tsconfigs.

3. **Added `/// <reference types="vitest/config" />` as the first line of `vite.config.ts`,
   and added `"vitest/globals"` to the `types` array in `tsconfig.app.json`.** With this
   toolchain's versions (Vite 8, Vitest 4, TypeScript 6), the literal Task 1 `vite.config.ts`
   content (`import { defineConfig } from 'vite'` + a `test: {...}` block) fails `tsc -b`
   with "Object literal may only specify known properties, and 'test' does not exist in
   type 'UserConfigExport'", and `src/App.test.tsx`'s bare `test`/`expect` globals fail with
   "Cannot find name 'test'"/"'expect'". `vitest run` itself (esbuild-transpile, not
   type-checked) was unaffected — `npm test` passed before this fix too — but `npm run build`
   (which the plan itself defines as `tsc -b && vite build`) would not have type-checked
   clean without it. Both fixes are additive-only: the triple-slash reference is the
   standard documented Vitest pattern for typing the `test` key without changing the
   `import` statement, and `vitest/globals` is Vitest's own ambient-types package for the
   `globals: true` runtime behavior already configured. No file's *specified* prose content
   was altered — only these two purely type-level, zero-runtime-effect additions were made
   on top of it.

4. Did not copy the Vite template's `README.md`; not part of the Task 1 file list.

None of these deviations changed runtime behavior of the app or the test; `npm test` and
`npm run build` both pass, and `npm run dev` serves the page without console errors.
